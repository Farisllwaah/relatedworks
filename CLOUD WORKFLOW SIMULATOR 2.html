1. Introduction
Cloud Workflow Simulator is a tool used to simulate workflow scheduling algorithms on
cloud infrastructure, based on the CloudSim framework. It is used to design algorithms which
can schedule a set of workflows with budget constraints and resource costs given. CWS takes
a set of workflows as input, then creates different ensembles from those workflows, and
finally performs simulations with provided parameters. Cloud Workflow Simulator 2 (CWS2)
is the next version of Cloud Workflow Simulator.
This document is a technical documentation for CWS’ developers. It helps to quickly
familiarize with the simulator architecture and its respective modules and to get to know how
to easily extend existing functionalities with new features. It is divided on following chapters:
• Chapter 2 presents general overview of the product including state-of-the-art of cloud
simulators.
• Chapter 3 contains requirements backlog. It is focused on features that were planned
to be introduced in CWS2, however it includes also a list of high-level requirements
that were less important and were planned to be implemented later.
• In Chapter 4 developer can find description of high-level simulator architecture and
see how it is modularized. Respective modules are thoroughly described in Chapter 5.
• Chapter 6 sets out predefined algorithms overview including their hierarchy and
architecture. Besides the algorithm module API chapter it is a good place to start for
algorithm developers.
• Chapter 7 introduces most important terms that are essential to understand the
codebase and problem domain. It does not cover terms that were listed in terms and
definitions chapter in user manual so it is better to start there.
• Chapter 8 consists of important assumptions that were made during development.
Simulators always use only a simplified model of reality and must make such
assumptions.
• Chapter 9 comprises code style guidelines for developers which they should obey in
order to keep the codebase clean and consistent.
• Chapter 10 is composed of some thoughts about the simulator design we came up
with during development but we had no time to implement them.

2. Product overview
This chapter contains an introduction to the Cloud Workflow Simulator 2. It consists of
product vision, description of users, list of team members and state of the art in the cloud
simulators research field.
2.1. Product vision
Cloud Workflow Simulator 2 is the next version of workflow aware cloud infrastructure
simulator (Cloud Workflow Simulator). The main goal of this project is to add new
functionalities to the existing product, as well as provide better user and developer experience
while interacting with it.
Currently, Cloud Workflow Simulator allows for running simulations of workflows on
modeled cloud infrastructure. The model consists of Virtual Machines, which imitate real
world counterparts by providing such parameters as cost-per-billing unit, provisioning time,
CPU power. Workflow’s tasks are scheduled by build in algorithms on the infrastructure and
the results are provided to the user. The results include parameters such as cost, time,
efficiency. Despite the fact that workflows include information about task’s input and output
files Cloud Workflow Simulator does model this part of the system, it just ignores file
transfers.
Cloud Workflow Simulator 2 will address the issue of file transfers unawareness. Storage
support will be added to the product to provide more realistic simulation environment. The
storage will be modeled in a way to allow for simulating popular global-state storages, such as
Amazon S3
1
or simple network file system. After the storage support is added, built in
algorithms will be updated to take file transfer information into account.
The next set of features to be introduced in Cloud Workflow Simulator 2 will enhance
user experience while running simulations. This specifically aims for creating scripts for
running, loading and preparing simulations both on personal computers and distributed grid
environments. Such set of tools will dramatically reduce the learning barrier which is required
now to start using Cloud Workflow Simulator.
Cloud Workflow Simulator currently does not provide any easy way to validate
algorithms and visualize simulation results. We aim to address this issue by creating
validations scripts, which will determine if the tested algorithm meets certain criteria. This

2.2. User description
Cloud Workflow Simulator 2 has three main types of end users. For all of them the main
activity is running simulations, but each of them has different purpose to do so. The types of
end users are:
• algorithm designers - This type of user is interested in assessing and designing
workflow scheduling algorithms. They use a set of workflows whose parameters are
well known among scientific community as an input. They analyze the outputs to
determine if assessed scheduling algorithm is performing well. Potentially, they repeat
this process many times while tuning up the algorithm.
• infrastructure testers - This type of user is primarily interested in assessing their own
cloud infrastructure. They run simulations taking as an input well known workflows
and standard algorithms. Then they analyze output from infrastructure provider
perspective to determine if the infrastructure is capable of running the given workflow
with the given parameters (e.g. algorithm type, budget, number of virtual machines).
• workflow testers - This type of user is primarily interested in testing their own
workflows. They run simulations using well known algorithms and infrastructure,
provided their own custom workflows. They analyze the outputs to assess tested
workflows and to decide if they are capable to be run on a real cloud infrastructure.

2.4. Cloud simulators state-of-the-art
CloudSim is a framework for modeling and simulation of cloud computing
infrastructures and services. Since it is a generic cloud simulation framework, it is rarely used
as a standalone product, in most cases it is used as a base for specific targeted simulators.
CloudSim hides underlying low level complexity of cloud infrastructures to allow developer
to focus on delivering features important in their simulations. Main features of CloudSim are
support for modeling and simulation of virtual machines with provisioning/deprovisioning
policies, energy-aware computational resources, network topologies and message passing
applications and federated clouds.
WorkflowSim is a simulator based on CloudSim. It specifically aims for workflow
simulations. It includes infrastructure for reading, parsing and processing workflow input
files. Although it seems that it is very similar to Cloud Workflow Simulator 2, it is different in
many scopes. First of all it supports only single workflows, whereas CWS2 is built for
simulating ensembles (sets of workflows). WorkflowSim also focuses on different areas of
experiments, such as heterogeneous distributed systems, various failure and delays models. It
models task queue which in batch scheduling systems can be of chief influence on simulation.
Comprehensive description of the simulator can be found in its original research paper [1].
GridSim is a grid simulation toolkit for resource modeling and application scheduling for
parallel and distributed computing. It was created before cloud computing term has been
invented, hence it does not refer to it. However, it uses concepts that are mostly valid in cloud
infrastructures. It mainly allows for modeling and simulation of grid resources for design and

Simulator 2 from its main feature perspective. GridSim supports heterogeneous computing
units which can have one or more processors and local or distributed memory. Resources can

congestion. There is a GridSim research paper available [2].
GreenCloud is a simulator primarily targeting for energy consumption simulations. It
operates on much lower conceptual lever than Cloud Workflow Simulator 2. It mainly focuses
on communication in the cloud and operates on network packet level events. It allows for
modeling and simulation of various data center IT resources, such as computing servers,
network switches and other communication links. GreenCloud can be used to develop and
simulate communication protocols, network infrastructures as well as resource allocation

3. Product requirements specification
This chapter describes requirement specification that we worked on during the course of
the project. We show here how we gathered requirements from our clients, and then list them
in form of user stories for functional requirements and simple bullet points for nonfunctional
requirements.
3.1. Specification description
Requirements specification was prepared using agile method called user stories. A user
story represents single requirement in a very concise way. We assumed one can always get
details from the client. A few high-level requirements called “epics” were distilled at the
beginning and we were incrementally expanding them into more detailed lower-level user
stores. We did not work on all of epics, conversely we focused only on those with the highest
priority.
Product backlog does not contain user stories for features that were already implemented
in CWS. It is a side effect of CWS2 implementation.
The backlog itself is ordered by the priority of user stories that our client gave us. In
order to prioritize them we used MoSCoW method. Ultimately only “must have” and “should
have” stories were considered for CWS2 but the rest was left for the sake of future
development.
From “must have” and “should have” stories we have created project backlog. We paid
more attention to those stories. We created detailed acceptance criteria that was validated by
client before we started working on CWS2.

5.2. Discrete event engine module
Message passing architecture
The simulation is performed using discrete event model. Objects are communicating with
each other using events like in a message passing distributed system. The only difference is
that when sending an event the object is declaring the interval this event is postponed for what
imitates real processes that take time like computation or file transfer. The module instead of
waiting for given time, changes the global time and executes the next event in the time-based
queue.

5.7. Global Storage module
Architecture
Global Storage is a module responsible for managing job’s input and output file transfers.
When a job is submitted for execution to a Virtual Machine it first needs to stage in
(download, fetch) required input files. This process happens when the Virtual Machine sends
an event saying that it is just before starting the job. Eventually, after all input files are staged
in the Global Storage informs the Virtual Machine about this fact. At this moment the
execution of the job happens. If the execution is successful, then the Virtual Machine send an
event informing about this fact. Then output files are staged out (uploaded) and Virtual
Machine is ready to broadcast that the job has successfully completed and uploaded its results.

When the Global Storage is informed about job’s finish or start it starts zero or more file
transfers (either stage in or stage out transfers). For each file separate transfer task is created.
For any given job all input/output transfers are started at the same time. When there are no
input/output files to transfer the process is considered successful instantaneously.
A transfer task has the following properties: the file identifier which is being transferred,
the file’s size in bytes, remaining size in bytes to transfer and duration in seconds. There is no
failure model for transfer tasks. This means that all transfers will eventually succeed, unless
the Virtual Machine of the job that caused the transfers is deprovisioned. Technically, transfer
do not run on any Virtual Machine, but they are executed on the Global Storage. However,
transfers are associated with job’s parent Virtual Machine.
The important question is why the Global Storage starts all job’s transfers at the same
time. The answer is the fact, that using this model files will be downloaded/uploaded no
slower than if the transfers were started sequentially. In the simplest case, when available
bandwidth is constant (which is not true with Global Storage, as will be described later), N
sequential transfers are completed in exactly the same time as N parallel transfers. This is
graphically shown on Figure 5.6.

and provisioning. There is also internal API for developers to incorporate their own
algorithms into Cloud Workflow Simulator 2. This chapter also shows the simulation model
that we have incorporated into the final product.
6.1. Scheduling algorithms state of the art
During the course of development of the project we have conducted studies of the state of
the art in the scheduling algorithms field. We have gathered our results here and divided them
into generic cloud scheduling algorithms and workflow aware ones.
6.2. Cloud scheduling algorithms
Scheduling individual tasks on the clouds is one of the most important duties of the
infrastructure providers. Having a good scheduler can lead to better resource utilization and
thus save large amount of money. Additionally, many public clouds emerged recently and
they have became easily accessible. All this factors encouraged research groups to tackle the
problem of individual task scheduling algorithms. Below we will describe the most important
publications from the perspective of our problem.
One of the first approaches to the scheduling task on clusters problem has been described
by Yair Amir et al in their paper “An Opportunity Cost Approach for Job Assignment and
Reassignment in a Scalable Computing Cluster”. Since it was published in early 2000s the
authors do not use term cloud, but the general idea is similar to what we call cloud today. In
the paper the authors want to schedule a set of task on a set of heterogeneous machines to
minimize the execution cost. They assume that every machine has only CPU and memory
which costs some amount of money and (what is unusual) the tasks can be migrated between
machines at any given time. There have been shown two improvements to scheduling
algorithms in such environments which lead to noticeable cost reductions.

6.3. Workflow aware scheduling algorithms
Workflow aware scheduling algorithms are often referenced as Direct Acyclic Graphs
(DAGs) scheduling algorithms. This problem is NP-complete in general, so most research
teams are applying various heuristics to the basic algorithms. Many of them achieve relatively
good results especially with real-life (non-synthetic) workflows as an input. There have been
many recent studies about this topic. Many of these studies implemented different
assumptions which changed the problem area or significantly reduced problem’s complexity.
The most notable approaches workflow aware scheduling algorithms will be described below.
L. F. Bittencourt and E. R. M. Madeira in their paper “A Cost Optimization Algorithm
for Workflow Scheduling in Hybrid Clouds” [8] describe and benchmark DAG scheduling
algorithm for hybrid clouds. By their definition, a hybrid cloud consists of a private cloud part
and public cloud part. The private cloud is considered to be slow and inflexible, but it costs
nothing to schedule a task there (i.e. there are some free resources for computing the
workflow). On the other hand, the public cloud is fast and has unlimited resources, but a user
has to pay for using that resources. Additionally, there is a maximum makespan (execution
time) parameter D. The researches introduce The Hybrid Cloud Optimized Cost algorithm
which schedules workflow on a hybrid cloud minimizing the computation cost while keeping
the makespan within the maximum boundaries. The algorithm’s benchmarks show that it
remarkably reduces computation cost (2 - 5 times) comparing to the other simple methods.
The topic of storage aware workflow scheduling is examined in the paper “Scheduling
Data-Intensive Workflows onto Storage-Constrained Distributed Resources” [9]. This was
one of the first attempts of solving this problem and produced results are very promising. The
authors wanted to schedule a data intensive workflow onto storage constrained infrastructure
(cloud). More than 50% of storage requirements reduction was achieved in the simulation of
sample, but real-life workflow. Additionally, a significant reduction of makespan was
observed. The general idea behind the scheduling algorithm consists of two phases. At first,
the algorithm adds transfer jobs to the workflow. These jobs are responsible for transferring
files from one resource (VM) to another. They are only created when needed, this is when two
dependant jobs are scheduled on different resources. This assumption is comprehensible and
easy to notice. The second phase allows for saving up resources. It creates cleanup jobs for
each file after detecting that it will be not needed anymore. After those two phases added jobs
are joined together using simple heuristics. The conclusion is that adding simple
enhancements for storage awareness to current algorithms can lead to noticeable reduction of
resource usage and as a result spending less money for the computations.

6.4. Predefined algorithms
Cloud Workflow Simulator 2 has a couple of built in scheduling algorithms. They are
intended to be used by people who are not interested in developing any new
scheduling/provisioning methods, but want to examine their infrastructure of workflows.
Predefined algorithms represent three different categories of scheduling/provisioning
algorithms to cover as much of problem space as possible.
Static Provisioning Static Scheduling (SPSS) algorithm statically provisions all necessary
Virtual Machines and statically creates scheduling plan. Statically means here, that all
computation is done before running a simulation, and when the simulation is started the
computes plan cannot change. This solution is handy when your case involves planning
before submitting jobs to the cloud infrastructure. Detailed description of the SPSS algorithm
can be found in the original Cloud Workflow Simulator research paper [3].
Dynamic Provisioning Dynamic Scheduling (DPDS) algorithm dynamically provisions
Virtual Machines and schedules jobs. It starts the simulation with initial number of Virtual
Machines and then schedules jobs. Then it inserts all admissible tasks into a workflow priority
based priority queue. A task is admissible if all tasks it depends have successfully executes or
it depends on zero tasks. Tasks from the priority queue are scheduled on arbitrarily chosen
available Virtual Machines. At any time, utilization of Virtual Machines is the percentage of
machines that are executing tasks. If at any time utilization is below lower threshold
(currently 70%) then DPDS starts deprovisioning Virtual Machines. If the utilization is
greater than upper threshold (currently 90%) then DPDS starts provisioning new Virtual
Machines. Detailed description of the DPDS algorithm can be found in the original Cloud
Workflow Simulator research paper [3].
Workflow Aware Dynamic Provisioning Dynamic Scheduling (WADPDS) algorithm
uses the same ideas as DPDS in terms of provisioning and scheduling. However, it uses
information about the structure of workflows when scheduling tasks. It automatically removes
all workflows that are impossible to schedule given the remaining time and budget. Detailed
description of the WADPDS algorithm can be found in the original Cloud Workflow
Simulator research paper [3].
6.5. Simulation model assumptions
There are a couple of important assumptions that have been made in order to make
simulations realistic and ensure that the model meets quality criteria.

Virtual Machines have deprovisioning and provisioning delays in seconds. This is the
latency between sending respective requests and executing them. Virtual Machines can be in
non-started, running and terminated states. Jobs can execute only on Virtual Machines in
running state. Only one job can execute on a Virtual Machine at the same time. Virtual
Machines have one CPU with computational power in MIPS.
Before a job is executed all its input files (if any) have to be staged in to the Virtual
Machine. Input files have to exist in the storage before staging them in. Jobs actual runtime is
based on Virtual Machine’s MIPS and its task’s predicted runtime. The actual runtime is
randomized under specified distribution. A job can finish successfully or fail. If a job fails it
can be then sent to any Virtual Machine for reexecution. Once a job finishes successfully it
stages out all its output files. Once all output files are staged out the environment is notified
about the fact that the job has finished.
A workflow is considered successfully finished if all its tasks are finished. Workflow’s
finish time is the finish time of the last job. Workflows have priority number assigned. The
lower the priority is the more important a workflow is. When a workflow with priority p
finishes it adds 2-p to the simulation score. The higher simulation score is the better. This
means that it is better to complete workflow with priority p than all other workflows with
smaller priority combined.

9. Conclusions
Cloud Workflow Simulator 2 is new version of an existing product. This caused that we
had to follow certain architectural patterns while working on the project. The legacy code
caused a lot of problems in the early stages of development. First of all, Cloud Workflow
Simulator was based on CloudSim framework. This framework requires programmers to
follow certain coding patterns which lead to hard-to-test code. What is more, Cloud Workflow
Simulator takes advantage of only small part of the framework (discrete event engine and a
couple of other things), which could be easily replaced by other solutions. Unfortunately, the
codebase was too big to resolve this problem when we started the project. Despite these
issues, the framework occurred to be easy to use and extend, so we have successfully finished
the project.
Choosing Python and Ruby for command line scripts was huge success, because both
languages are easier to code in than shell scripts. Separating certain logic from the core
product package to separate script allowed for flexibility. This advantage was clearly visible
when we started testing the product on remote grid environments, which require programs to
follow patterns very different from common personal computers. Additionally, separating
validation and visualization scripts from Cloud Workflow Simulator 2 gave us the possibility
to easily create different running flows for different use cases.
We decided to implement storage awareness in completely new module. This was good
idea, because we were able to achieve high test coverage for this particular module, to write
code that has one well defined responsibility and was easily replaceable. Thanks to this, we
implemented two strategies for storage awareness. The first one was Global Storage (as
described in earlier chapters) and Void Storage which basically ignores file transfers.
Our final project goal was to completely rewrite scheduling algorithms API. The main
intention was to make the API generic, extensible and flexible. We did not manage to do this,
because we have further focused on features more important from the client’s perspective.
However, we have conducted comprehensive analysis and planning of what has to be done in
order to achieve this goal. We hope that we will have an opportunity to finish this task in the
future.
